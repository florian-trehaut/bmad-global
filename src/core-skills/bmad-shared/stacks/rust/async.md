---

Rust's `async/await` is a zero-cost, compiler-desugared state-machine model — there is no runtime in the language itself. Every async program picks **one** executor (tokio is the default for networking; embassy for embedded; smol for minimal). This file catalogues the patterns reviewers MUST flag when reading async Rust code in May 2026.

## Async fundamentals

- **`async fn` returns `impl Future<Output = T>`** — the compiler desugars the function body into an anonymous state-machine struct that implements `Future`. The body does not run until polled. `let f = my_async_fn();` does nothing on its own; `f.await` is what drives it. This is the #1 source of "my logging never appears" bugs — forgetting the `.await` produces a `must_use` warning but rarely an error.
- **`.await` is the only yield point.** Between two `.await` calls the code is purely synchronous and CANNOT be interrupted by the executor — long CPU-bound stretches between awaits block the entire worker thread (cross-ref `tokio::task::spawn_blocking` below). Yielding manually for fairness: `tokio::task::yield_now().await` returns control to the scheduler at a chosen point.
- **Poll model.** The runtime calls `Future::poll(&mut self, cx: &mut Context<'_>) -> Poll<T>`. `Poll::Pending` means "register the waker and try again later"; `Poll::Ready(value)` means "done". The full `Future` trait:

  ```rust
  pub trait Future {
      type Output;
      fn poll(self: Pin<&mut Self>, cx: &mut Context<'_>) -> Poll<Self::Output>;
  }
  ```

- **`Waker`.** The reactor (epoll/kqueue/io_uring) calls `cx.waker().wake()` when the resource the future is parked on becomes ready (socket readable, timer expired, channel non-empty). Hand-rolled futures MUST register the waker before returning `Pending` or they deadlock silently — the executor will never re-poll a task whose waker was never armed.
- **`Pin<P>`.** Futures generated by `async fn` are self-referential (the state machine stores both an `&Foo` and the `Foo` it points to in the same struct). `Pin<&mut T>` guarantees the value never moves in memory so the internal references stay valid. The compiler hides `Pin` for `async fn`; manual `Future` impls and recursive async fns must deal with it explicitly. Stack-pin via `std::pin::pin!(future)` (stable since 1.68); heap-pin via `Box::pin(future)`.
- **`Send` / `Sync` bounds.** A future is `Send` iff every value held across an `.await` is `Send`. `tokio::spawn` requires `Future: Send + 'static`. Holding `Rc<T>`, `RefCell<T>`, or a `MutexGuard<'_, T>` across an await silently makes the future `!Send` — compilation only fails at the `spawn` site, often far from the actual offending line. Use `tokio::task::spawn_local` inside a `LocalSet` if you genuinely need `!Send` futures (e.g., GUI thread interop).
- **Cooperative scheduling.** Tokio's `current_thread` runtime polls one task at a time on one thread; the `multi_thread` runtime work-steals across N worker threads (default = num CPUs). Neither pre-empts: a hot loop without `.await` will starve sibling tasks until the next yield point. Tokio's automatic yield budget (since 1.32) inserts a yield every 128 polls of a task to reduce starvation, but it is not a substitute for designing yield points.
- **`'static` lifetime requirement.** `tokio::spawn`, `JoinSet::spawn`, and any spawn into a long-lived runtime require `Future: 'static`. The bound exists because the runtime may schedule the task after the calling stack frame has returned; non-`'static` borrows would be use-after-free. Workaround = `Arc<T>` for sharing, owned values via `clone()`/`move`, or scoped concurrency (`tokio::task::JoinSet` awaited before returning).

Cite: [Async Book — Under the hood](https://rust-lang.github.io/async-book/02_execution/01_chapter.html), [std::future](https://doc.rust-lang.org/std/future/index.html), [Tokio Tutorial — Async in depth](https://tokio.rs/tokio/tutorial/async).

## async fn in traits (Rust 1.75+)

- **Stabilized in Rust 1.75 (Dec 2023):** `async fn` may appear directly in trait definitions and impls. Same release also stabilized return-position `impl Trait` in traits (RPITIT). [Announcing async fn and RPITIT — Rust Blog](https://blog.rust-lang.org/2023/12/21/async-fn-rpit-in-traits/).
- **Rust 1.85 (Feb 2025):** added async closures `async || { ... }` returning futures while capturing local state. Not the same as async fn in trait — distinct feature. [Rust 1.85.0 release notes](https://blog.rust-lang.org/2025/02/20/Rust-1.85.0/).
- **Caveat — no `dyn` dispatch.** `dyn Trait` where the trait has an `async fn` does NOT work natively — the returned `impl Future` has a different concrete type per impl, so the vtable layout is undefined. The workaround is `#[async_trait]` (returns `Pin<Box<dyn Future + Send + 'async_trait>>` via macro expansion) or manually writing `fn foo(&self) -> Pin<Box<dyn Future<Output = T> + Send + '_>>`.
- **Caveat — Send bounds are sticky.** A `trait Foo { async fn bar(&self); }` returns a future that is `Send` only if every awaited value is `Send`. When you call `bar` through a generic `T: Foo`, the compiler cannot prove the future is `Send` without an explicit bound. Common idiom: declare a separate `trait FooSend: Foo where for<'a> Self::Bar<'a>: Send` (via GAT) or use the `#[trait_variant::make(Send)]` attribute (added 2024).
- **`async-trait` crate** — still the de-facto solution for `dyn Trait` async, older MSRVs, and when the Send-bounds dance is unworkable. The macro boxes every call, so it has a small heap cost per invocation; profile before assuming it's a hotspot. <https://docs.rs/async-trait>.

## Anti-patterns to flag

- **Mixing two runtimes in the same crate** (BLOCKER)
  - Detection: `Cargo.toml` has both `tokio` and `async-std` (or `tokio` and `smol`, etc.) as runtime features. Or `#[tokio::main]` in `main.rs` while a dependency exposes a `smol::block_on` API. Run `cargo tree --duplicates` looking for `tokio` AND `async-io` / `async-executor` co-occurring.
  - Why: each runtime carries its own reactor (epoll loop, timer wheel, blocking pool). A `tokio::sleep` future created in scope A and awaited in `async_std::task::block_on` will park forever because the tokio reactor was never started. Same hazard with mixed timers / readers / mutexes. The bug looks like a deadlock but is actually "waker never fires".
  - Fix: pick one runtime per binary. If a library exposes `async fn`, it should be runtime-agnostic (only depend on `futures` + `std::future`); if it must spawn, accept a spawner trait, feature-gate the runtime choice, or use `agnostik`/`async-executor` as a portability shim. Forbid mixed runtimes in CI via `cargo deny` rules.

- **Blocking I/O or sleep in async context** (BLOCKER)
  - Detection: `std::thread::sleep`, `std::fs::read_to_string`, `reqwest::blocking::*`, sync DB drivers (rusqlite without `tokio-rusqlite`, diesel without `diesel-async`), `Mutex::lock` on huge contention paths — all called from inside an `async fn` without `spawn_blocking`. CPU-bound work (hashing, image resize, compression) over a few microseconds belongs to the same anti-pattern even though it's not "I/O".
  - Why: each blocks the worker thread for the full duration. On `current_thread` runtime this halts EVERYTHING; on `multi_thread` it locks up 1/N of capacity. A request stuck on a 30-second sync DB call freezes every task scheduled on that worker. Tokio's automatic yield budget does NOT save you here — it only inserts yields between polls of THIS task; if the task never yields, the worker thread is gone.
  - Fix: use the async equivalent (`tokio::time::sleep`, `tokio::fs::read`, `reqwest` async API, `sqlx`); for unavoidable sync code, wrap in `tokio::task::spawn_blocking(move || { sync_work() }).await?`. `spawn_blocking` runs on a dedicated thread pool (default 512 threads, configurable). For parallel CPU work, prefer `rayon::spawn` + `tokio::sync::oneshot` to bridge the result back to the async world.

- **`std::sync::MutexGuard` held across `.await`** (BLOCKER)
  - Cross-ref: full description in `stacks/rust/concurrency.md`. Listed here because it is the single most common tokio production deadlock and lint `clippy::await_holding_lock` catches it.

- **Spawning futures with non-`'static` borrows via `unsafe`** (BLOCKER)
  - Detection: `unsafe { tokio::spawn(future_with_borrow) }`, `mem::transmute` to extend lifetimes, hand-rolled `Pin<Box<dyn Future>>` casts that erase a borrow lifetime.
  - Why: `tokio::spawn` requires `'static` because the runtime may execute the task after the borrowed value is dropped. Bypassing the bound is undefined behaviour — a use-after-free disguised as an async bug, undetectable by `cargo test`.
  - Fix: clone the borrowed data (or wrap in `Arc<_>`), use scoped concurrency primitives instead (`tokio::task::JoinSet` polled in a function that outlives the borrow; `async_scoped` crate; or restructure to take ownership).

- **`tokio::spawn(future).await` where the JoinHandle is then dropped** (MAJOR)
  - Detection: `tokio::spawn(...)` whose `JoinHandle` is bound to `_` or never awaited, while the task references stack-borrowed resources or its completion is required.
  - Why: dropping a `JoinHandle` does NOT cancel the task (it detaches). But ignoring it means panics inside the task become silent `JoinError`s and Result-returning tasks lose their result. Tests that "pass" because the assertion ran inside a detached task that already panicked.
  - Fix: bind the handle, await it, propagate via `handle.await??` (the double `?` unpacks `JoinError` then the inner `Result`). Or use `JoinSet` for batches.

- **`.unwrap()` on `JoinHandle::await`** (MAJOR)
  - Detection: `handle.await.unwrap()` in production code paths.
  - Why: `JoinHandle::await` returns `Result<T, JoinError>` where `JoinError` covers `panic` and `cancelled`. `unwrap()` here panics the parent task on any child panic — which usually kills the whole runtime. Cross-ref `stacks/rust/null-safety.md` for the general unwrap rule.
  - Fix: `match handle.await { Ok(v) => ..., Err(e) if e.is_panic() => log::error!("task panicked: {e}"), Err(e) => log::warn!("task cancelled: {e}") }`. Or `handle.await??` if the inner `T` is itself a `Result`.

- **`tokio::sync::Mutex` used everywhere "to be safe"** (MAJOR)
  - Detection: `tokio::sync::Mutex<T>` for data where the critical section contains no `.await`.
  - Why: `tokio::sync::Mutex` is a fairness-queueing async mutex — every `lock().await` allocates a waiter node and may yield to the executor. `std::sync::Mutex` or `parking_lot::Mutex` are ~3-5x faster when contention exists and the critical section is short. The "safe" choice is the slow one.
  - Fix: default to `std::sync::Mutex` / `parking_lot::Mutex`; only escalate to `tokio::sync::Mutex` when the critical section MUST hold across an `.await` (e.g., serialised I/O against a single connection).

- **Recursion in `async fn` without `Box::pin`** (MAJOR — usually compile error, but design smell)
  - Detection: `async fn walk(node: Node) -> () { for c in node.children { walk(c).await } }`.
  - Why: the desugared state machine type contains itself, which would be infinitely sized. Since Rust 1.77 the compiler emits a helpful error suggesting `Box::pin` indirection; before 1.77 it was a cryptic `E0733`.
  - Fix: `fn walk(node: Node) -> Pin<Box<dyn Future<Output = ()> + Send + '_>> { Box::pin(async move { for c in node.children { walk(c).await; } }) }`. Or use the `async_recursion` crate for cleaner syntax: `#[async_recursion] async fn walk(node: Node) { ... }` — the macro inserts the `Box::pin` for you. Watch the depth: each recursive call allocates a `Box` on the heap; deeply recursive traversal over user-controlled tree depth is a memory-amplification vector.

- **`tokio::join!` for many or dynamic futures** (MINOR)
  - Detection: `tokio::join!(a, b, c, d, e, f, ...)` with > 4 branches, or `join!` driving a `Vec<Future>` via an unrolled macro call.
  - Why: `join!` polls all branches on the current task — losing concurrency on CPU-bound futures and inflating the future size. For dynamic counts you cannot use it at all.
  - Fix: `futures::future::join_all(vec)`, `futures::stream::iter(vec).buffer_unordered(N).collect().await`, or `tokio::task::JoinSet` for dynamic spawn-then-await.

- **`Box::pin(async { ... })` for stack-allocatable futures** (MINOR)
  - Detection: `Box::pin(async move { ... })` immediately consumed by `.await` in the same scope.
  - Why: each `Box::pin` allocates on the heap. For non-recursive, non-`dyn` use cases the compiler can keep the future on the stack via `pin!`.
  - Fix: `let fut = std::pin::pin!(async { ... }); fut.await;` (macro stabilised since Rust 1.68).

- **`println!` / `eprintln!` in async hot paths** (MINOR)
  - Detection: `println!("got request: {:?}", req)` inside an async handler.
  - Why: `println!` takes a global stdout lock and performs synchronous I/O — blocks the worker thread for the duration of the write. Under load this serialises requests behind stdout.
  - Fix: `tracing::info!(?req, "got request")` with a non-blocking writer (`tracing_appender::non_blocking`). Cross-ref the "Tracing & observability" section below.

- **Unbounded `tokio::spawn` from external input** (MAJOR — cross-ref `stacks/rust/concurrency.md` for full detection/fix; relisted because it's specifically an async-runtime trap).

- **`futures::executor::block_on` inside tokio context** (MAJOR)
  - Detection: a sync function called from `async` code internally calls `futures::executor::block_on(...)` or `tokio::runtime::Handle::current().block_on(...)` on a future.
  - Why: `block_on` from the futures crate spins a tiny in-place executor that knows nothing about the surrounding tokio runtime — timers don't fire, the tokio reactor is paused on this thread. `Handle::block_on` will outright panic from within an async context ("Cannot start a runtime from within a runtime"). Even when it does not panic, the thread stops running other tokio tasks until completion.
  - Fix: convert the surrounding code to `async` and `.await` directly; or hop to a dedicated blocking pool via `tokio::task::spawn_blocking(move || futures::executor::block_on(future))`.

- **Forgetting `tokio_unstable` for `console-subscriber`** (MINOR)
  - Detection: `console_subscriber::init()` present but `RUSTFLAGS` lacks `--cfg tokio_unstable`. Either compile fails or `tokio-console` shows no tasks.
  - Why: tokio-console depends on unstable tokio internals (per-task tracing IDs, runtime metrics) gated behind that cfg flag.
  - Fix: in `.cargo/config.toml`, `[build] rustflags = ["--cfg", "tokio_unstable"]`.

- **`tokio::main` on a library crate** (MAJOR)
  - Detection: a crate of type `lib` exposes `pub async fn run() -> ...` and applies `#[tokio::main]` to it.
  - Why: `#[tokio::main]` turns the function into a sync `fn` that creates a runtime — invalid inside a library because callers already have their own runtime. The runtime-creation panic happens at the call site, not at compile time.
  - Fix: libraries expose `pub async fn` and let the binary pick the runtime. If a library MUST own its runtime (e.g., it embeds tokio-uring while the host uses tokio), spawn an internal worker thread and communicate via channels.

- **Async I/O without timeouts on external dependencies** (MAJOR)
  - Detection: `reqwest::get(url).await?` (or `tokio::net::TcpStream::connect`, sqlx queries, etc.) without `tokio::time::timeout` or per-client timeout configuration.
  - Why: a stuck upstream hangs the request forever. Multiplied by traffic, you exhaust connection pools, exhaust task slots, and brown-out the whole service. "It works in tests" because the local mock answers in milliseconds.
  - Fix: configure timeouts at the client level (`Client::builder().timeout(Duration::from_secs(5))` for reqwest; `connect_timeout` + `acquire_timeout` for sqlx) AND wrap critical paths in `tokio::time::timeout(d, fut).await??`. Make timeout values explicit constants, not magic numbers.

## Required guardrails

- **One runtime per crate.** `tokio` is default; document deviations in the project's `project.md#tech-stack` section. CI MUST forbid mixed runtimes via `cargo deny` or a custom check that fails when both `tokio` and `async-std` (or `smol`) appear in `Cargo.lock`.
- **Clippy lints enabled in `lib.rs`:**
  - `#![warn(clippy::await_holding_lock)]` — catches sync `MutexGuard` across `.await`.
  - `#![warn(clippy::await_holding_refcell_ref)]` — same hazard for `RefCell`.
  - `#![warn(clippy::async_yields_async)]` — flags `async fn` that returns another `Future` instead of awaiting it.
  - `#![warn(clippy::large_futures)]` — warns when a future's size exceeds 16 KiB (tunable).
- **Tracing crate, not `log` crate.** `tracing` carries span context across `.await` (via `.instrument(span)`); `log` macros do not. Cross-ref Tracing section.
- **`tokio-console` available for production debugging.** `console-subscriber = "0.4"` as an opt-in feature; running `tokio-console` against `localhost:6669` reveals task busy/idle ratios, lock contention, and dropped futures. <https://github.com/tokio-rs/console>.
- **`#[tokio::main(flavor = "multi_thread")]` for production binaries**, `#[tokio::main(flavor = "current_thread")]` for tests that don't need parallelism (faster, deterministic). `#[tokio::test]` for async unit tests; `#[tokio::test(flavor = "multi_thread", worker_threads = 4)]` when concurrency is part of the contract under test.
- **Cargo features explicit, not `full`.** `features = ["full"]` pulls in every tokio module — slower compile, bigger binary. List only what you use.
- **CI `cargo clippy --all-targets --all-features -- -D warnings`** treats lints as errors. Pair with `cargo clippy --workspace` so workspace members are checked too.

## Language-specific principles

- **`async fn` is not magic threading.** The function is a state machine; concurrency only happens when you `spawn` or combine futures (`join!`, `select!`, `JoinSet`, `buffer_unordered`).
- **Spawn requires `'static`.** Plan ownership upfront — `Arc<T>` for shared immutable, `Arc<Mutex<T>>` for shared mutable, owned clone otherwise. Trying to spawn with a borrow is the #1 reason juniors reach for `unsafe`.
- **`Send` and `Sync` are design constraints, not bureaucracy.** Decide at design time whether your futures cross thread boundaries. Reworking a non-`Send` graph deep into a project is painful.
- **Cancellation safety is a design property.** Every future that may sit in a `tokio::select!` branch or a timeout MUST be safe to drop mid-`.await`. See dedicated section.
- **Async is cooperative.** Yield ONLY at `.await` points. A 200ms hash loop between two awaits stalls every sibling task on the worker. Move CPU-bound work into `tokio::task::spawn_blocking` (for sync code) or `rayon` (for parallel CPU work).
- **`Pin` is for self-references.** Most code never sees it directly. When writing manual `Future` impls or self-referential generators, pinning rules apply — read the [pin module docs](https://doc.rust-lang.org/std/pin/) once and keep them open.
- **Prefer `JoinSet` over `Vec<JoinHandle<T>>`.** `JoinSet` handles dynamic add/remove, automatic abort on drop, and `join_next` returns completions in arrival order. <https://docs.rs/tokio/latest/tokio/task/struct.JoinSet.html>.
- **`tokio::select!` is for cancellation, not just concurrent waiting.** When a branch wins, the other futures are dropped — so design those futures to be cancellation-safe (see below).
- **Test async code with `tokio::test`, never `block_on` inside a sync `#[test]`.** The macro sets up a runtime correctly and integrates with `cargo test`'s threading model.
- **`tokio::time::sleep` is cancellation-safe; `tokio::time::interval` may yield bursts after lag.** Use `interval.set_missed_tick_behavior(MissedTickBehavior::Skip)` to avoid bursty ticks after a backpressure stall.

## Cancellation safety

A future is **cancellation-safe** iff dropping it mid-`.await` leaves the surrounding state machine in a valid, recoverable state. Cancellation happens implicitly:

- `tokio::select!` drops the losing branches when the winner completes.
- `tokio::time::timeout(d, fut).await` drops `fut` when the timer fires first.
- `tokio::task::JoinHandle::abort()` cancels the spawned task at the next yield.
- A `tokio::spawn`ed task is dropped if the `JoinSet` it belongs to is dropped (no `await`).

| Future                                     | Cancel-safe?       | Notes                                                                                                                                                                                                                                                                            |
| ------------------------------------------ | ------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `tokio::time::sleep`                       | YES                | Drop just discards the timer entry.                                                                                                                                                                                                                                              |
| `tokio::sync::Notify::notified()`          | YES                | The waiter is removed on drop.                                                                                                                                                                                                                                                   |
| `tokio::sync::mpsc::Receiver::recv`        | YES                | The next call sees the same queue.                                                                                                                                                                                                                                               |
| `tokio::sync::mpsc::Sender::send`          | **NO**             | If dropped after the value entered the channel buffer but before the receiver took it, the value is delivered — but if dropped _before_ enqueuing, the value is lost. Use `reserve()` first, then `send_permit.send(value)` (the `Permit` makes the actual transfer infallible). |
| `tokio::sync::broadcast::Receiver::recv`   | YES (with caveats) | Cancel-safe per docs; may skip messages on lag.                                                                                                                                                                                                                                  |
| `tokio::net::TcpStream::read` / `read_buf` | **NO**             | A partial read consumed from the socket is lost on drop.                                                                                                                                                                                                                         |
| `tokio::net::TcpStream::write_all`         | **NO**             | Partial write may have made it to the kernel buffer; retrying writes duplicate bytes.                                                                                                                                                                                            |
| `tokio::fs::File::read_to_end`             | **NO**             | Same partial-read issue.                                                                                                                                                                                                                                                         |
| `tokio::process::Child::wait`              | YES                | Status is re-readable on next call.                                                                                                                                                                                                                                              |

**Patterns:**

- **`tokio::pin!` to reuse a future across loop iterations** so the same in-progress state survives a `select!` branch loss:

  ```rust
  let sleep = tokio::time::sleep(Duration::from_secs(5));
  tokio::pin!(sleep);
  loop {
      tokio::select! {
          _ = &mut sleep => break,
          msg = rx.recv() => handle(msg).await,
      }
  }
  ```

  Without `pin!` + `&mut sleep`, each loop iteration would create a fresh 5s sleep — the timer would never fire.

- **`CancellationToken` (from `tokio_util::sync`)** for cooperative shutdown across many tasks. Clone the token freely; calling `.cancel()` on any clone wakes every waiter. Pattern for a worker:

  ```rust
  use tokio_util::sync::CancellationToken;
  async fn worker(token: CancellationToken, mut rx: Receiver<Job>) {
      loop {
          tokio::select! {
              _ = token.cancelled() => break,
              Some(job) = rx.recv() => process(job).await,
              else => break,  // channel closed
          }
      }
      // graceful shutdown: drain in-flight work, flush metrics, etc.
  }
  ```

  Reference: <https://docs.rs/tokio-util/latest/tokio_util/sync/struct.CancellationToken.html>.

- **`tokio::sync::mpsc::Sender::reserve()` then `permit.send(value)`** to make the actual transfer infallible and cancel-safe — the value never sits half-sent:

  ```rust
  let permit = sender.reserve().await?;  // .await may yield; cancel-safe
  permit.send(value);                     // infallible, synchronous
  ```

- **Critical sections drop guards before `.await`** — even cancellation-safe futures shouldn't hold locks. Cross-ref `stacks/rust/concurrency.md`.

- **Graceful shutdown with signal handling:**

  ```rust
  #[tokio::main]
  async fn main() -> anyhow::Result<()> {
      let token = CancellationToken::new();
      let server_task = tokio::spawn(run_server(token.clone()));

      tokio::select! {
          _ = tokio::signal::ctrl_c() => tracing::info!("SIGINT received"),
          _ = tokio::signal::unix::signal(SignalKind::terminate())?.recv() => {
              tracing::info!("SIGTERM received");
          }
      }
      token.cancel();
      server_task.await??;
      Ok(())
  }
  ```

  Cross-ref [Tokio — Graceful Shutdown](https://tokio.rs/tokio/topics/shutdown).

- **Document cancel-safety in your APIs.** Every `pub async fn` you ship in a library SHOULD declare in its rustdoc whether it is cancel-safe (and what cancellation means semantically). [Tokio `select!` cancellation-safety table](https://docs.rs/tokio/latest/tokio/macro.select.html#cancellation-safety) is the canonical reference. Oxide's [RFD 400](https://rfd.shared.oxide.computer/rfd/0400) is a real-world case study of designing cancel-safety into a fleet-management service.

## Error propagation in async

- `async fn foo() -> Result<T, E>` desugars to `fn foo() -> impl Future<Output = Result<T, E>>`. The `?` operator works inside `async` blocks identically to sync code:

  ```rust
  async fn pipeline(id: UserId) -> Result<Report, AppError> {
      let user = fetch_user(id).await?;
      let posts = fetch_posts(user.id).await?;
      Ok(summarise(user, posts))
  }
  ```

- **`tokio::spawn` returns `JoinHandle<T>`**, not `JoinHandle<Result<T, E>>`. The task can:
  1. Return `Ok(value)` — `handle.await => Ok(Ok(value))`.
  2. Return `Err(e)` — `handle.await => Ok(Err(e))`.
  3. Panic — `handle.await => Err(JoinError { panic: true, .. })`.
  4. Be aborted — `handle.await => Err(JoinError { cancelled: true, .. })`.
     Two `?` operators unpack both layers: `let v: T = handle.await??;`. Explicit match required when you want to distinguish panic vs cancellation:

  ```rust
  match handle.await {
      Ok(Ok(v)) => /* success */,
      Ok(Err(e)) => /* business error */,
      Err(e) if e.is_panic() => /* task panicked */,
      Err(e) if e.is_cancelled() => /* task aborted */,
      _ => unreachable!(),
  }
  ```

- **`tokio::time::timeout(d, fut)`** returns `Result<T, Elapsed>` if the inner future is itself `T`. With a `Result`-returning future you get `Result<Result<T, E>, Elapsed>` — flatten via `??` (and a `From<Elapsed> for E` impl), `transpose()`, or explicit match. Pattern: define a single `TimeoutError(#[from] Elapsed)` variant in your error enum so `??` just works.
- **Library boundary:** use `thiserror`-derived concrete enums so callers can match variants. <https://docs.rs/thiserror>. Example:

  ```rust
  #[derive(thiserror::Error, Debug)]
  pub enum DbError {
      #[error("query timed out after {0:?}")]
      Timeout(Duration),
      #[error("connection pool exhausted")]
      PoolExhausted,
      #[error(transparent)]
      Sqlx(#[from] sqlx::Error),
  }
  ```

- **Application boundary (handlers, main, CLIs):** use `anyhow::Result<T>` with `.context("loading config")` / `.with_context(|| format!("user_id={user_id}"))` to attach human-readable trails. <https://docs.rs/anyhow>.
- **`eyre`** is the same idea as `anyhow` with custom reporter hooks (e.g., `color-eyre` for pretty panic dumps in dev, `tracing-error::SpanTrace` for span context in error chains). Use when you want structured backtraces tied to tracing spans.
- **Async error context is critical** — without `.context(...)` an error that bubbled through six awaits says only "permission denied" with no trail of which file, which user, which request. Annotate at every async boundary. Combined with `#[instrument]` spans, error context + span trail makes debugging a 3am pager call tractable.
- Cross-ref `stacks/rust/null-safety.md` for the `unwrap()` rule and `?`-versus-`match` choice.

## Streams & async iteration

- **`futures::Stream`** is the async analogue of `Iterator`. `poll_next(cx) -> Poll<Option<Item>>` returns `Ready(Some(_))`, `Ready(None)` (terminated), or `Pending`. <https://docs.rs/futures/latest/futures/stream/trait.Stream.html>.
- **`StreamExt`** provides combinators: `.map`, `.filter`, `.then`, `.collect`, `.next`, `.fold`, and crucially `.buffer_unordered(N)` / `.buffered(N)` for bounded-concurrency I/O fan-out. <https://docs.rs/futures/latest/futures/stream/trait.StreamExt.html>.
- **`tokio_stream`** offers tokio-specific extensions: `wrappers::ReceiverStream` (channel → Stream), `StreamMap` (key-tagged multiplexing), `StreamExt::timeout` (per-item timeout).
- **`async_stream::stream! { yield x; }`** macro turns generator-like syntax into a `Stream` impl. Useful for state machines that need to `yield` between awaits. <https://docs.rs/async-stream>.
- **`for await x in stream` is not stable** as of May 2026. Use:

  ```rust
  use futures::StreamExt;
  let mut s = make_stream();
  while let Some(item) = s.next().await {
      handle(item).await?;
  }
  ```

- **Backpressure** is built into bounded channels. `tokio::sync::mpsc::channel(N)` returns a `(Sender, Receiver)` where `Sender::send` `await`s when the buffer is full — natural backpressure with no extra work. Unbounded variants (`unbounded_channel`) shift the problem to RAM and OOM kills.
- **Concurrent stream processing pattern:**

  ```rust
  use futures::stream::{self, StreamExt};
  let results: Vec<_> = stream::iter(items)
      .map(|item| async move { process(item).await })
      .buffer_unordered(16)
      .collect()
      .await;
  ```

  `buffer_unordered(16)` keeps 16 futures in flight; `buffered(16)` preserves order (slower because of head-of-line blocking). For Kafka-style consumer loops, prefer `buffer_unordered` + a downstream commit step that re-orders if needed.

- **`StreamExt::timeout(duration)`** (from `tokio_stream::StreamExt`) wraps each item in a per-item timeout — useful for heartbeat detection on long-lived streams.
- **Async-stream macro for ergonomic generators:**

  ```rust
  use async_stream::stream;
  fn paginate(base_url: String) -> impl Stream<Item = Result<Page, Error>> {
      stream! {
          let mut cursor = None;
          loop {
              let page = fetch(&base_url, cursor).await?;
              cursor = page.next_cursor.clone();
              let done = cursor.is_none();
              yield Ok(page);
              if done { break; }
          }
      }
  }
  ```

- Cite: [Rust Book Ch. 17 — Streams](https://doc.rust-lang.org/book/ch17-04-streams.html), [futures::stream::StreamExt](https://docs.rs/futures/latest/futures/stream/trait.StreamExt.html).

## Async I/O patterns

- **`tokio::fs`** wraps blocking `std::fs` calls in `spawn_blocking` internally. There is no POSIX async file I/O without io_uring; tokio uses the blocking pool by default. For Linux high-throughput file I/O, switch to `tokio-uring` or `monoio`.
- **`tokio::net`** — `TcpListener` / `TcpStream` / `UdpSocket` / `UnixStream` (Unix only). Pair with `AsyncReadExt` / `AsyncWriteExt` traits for ergonomic `read` / `write_all` / `read_to_string`.
- **`hyper`** is the low-level HTTP/1.1 + HTTP/2 + HTTP/3 (via h3 crate) library, used internally by reqwest, axum, warp, and tonic. Direct use mostly when writing custom proxies. <https://hyper.rs>.
- **`reqwest`** — ergonomic async HTTP client. Async by default (`reqwest::Client`), blocking opt-in (`reqwest::blocking::Client`) is a separate API. `rustls` TLS backend recommended over `native-tls` for static binaries and reproducible builds. <https://docs.rs/reqwest>.
- **`axum` 0.8** (Mar 2026) — handler-oriented framework on hyper + tower + tokio. Type-driven extractors (`State`, `Path`, `Query`, `Json`), tower middleware reuse, fastest-growing Rust web framework by download count in 2026. <https://docs.rs/axum>.
- **`actix-web`** — alternative framework, actor-model under the hood. Mature ecosystem, slightly different middleware shape; both axum and actix are production-grade. <https://docs.rs/actix-web>.
- **`tonic`** — gRPC framework on hyper + tower + tokio + prost. Code generation from `.proto` via `tonic-build`. <https://github.com/hyperium/tonic>.
- **`sqlx` 0.8+** — async SQL toolkit with compile-time-checked queries (`sqlx::query!` macro validates against a live DB at build time). Supports PostgreSQL, MySQL, SQLite, MSSQL. <https://docs.rs/sqlx>.
- **Connection pooling:** `sqlx::PgPool::connect_with(opts)` for SQL; `deadpool-redis` / `bb8-redis` for Redis; `deadpool-postgres` if you prefer the `tokio-postgres` driver layer over sqlx.
- **HTTP/2 server push, HTTP/3 / QUIC:** `quinn` (QUIC) + `h3` (HTTP/3 over QUIC) for low-latency edge stacks; `hyper`'s HTTP/3 support is feature-gated behind `h3`.
- **WebSocket:** `tokio-tungstenite` (client + server), or axum's `ws` extractor (built on tungstenite).

## Testing async code

- **`#[tokio::test]`** — transforms an `async fn` test into a sync `#[test]` that drives a current-thread runtime. Default flavor is `current_thread`; pass `flavor = "multi_thread", worker_threads = N` to test concurrency contracts:

  ```rust
  #[tokio::test]
  async fn test_handler_serves_request() {
      let response = handler(make_request()).await.unwrap();
      assert_eq!(response.status(), 200);
  }
  ```

- **`#[tokio::test(start_paused = true)]`** — starts with virtual time paused. Combined with `tokio::time::advance(d)` it makes time-dependent tests deterministic and instant. Requires feature `test-util`:

  ```rust
  #[tokio::test(start_paused = true)]
  async fn timeout_fires_after_5s() {
      let fut = tokio::time::timeout(Duration::from_secs(5), never_resolves());
      tokio::pin!(fut);
      tokio::time::advance(Duration::from_secs(4)).await;
      assert!(futures::poll!(&mut fut).is_pending());
      tokio::time::advance(Duration::from_secs(2)).await;
      assert!(fut.await.is_err());
  }
  ```

- **`tokio::time::timeout` in every async test.** Wrap the test body in `tokio::time::timeout(Duration::from_secs(10), async { ... }).await??` to convert hangs into hard failures with stack traces. Prevents CI deadlocks.
- **`wiremock`** — async HTTP mock server with random port (parallel-friendly), regex / JSON path matchers, request verification on drop. <https://docs.rs/wiremock>.
- **`mockito`** — alternative HTTP mock, simpler API, currently catching up on `wiremock`'s features.
- **`mockall`** — trait-level mocks via macro. Works with `#[async_trait]` and Rust 1.75+ async fn in trait (with `mockall` ≥ 0.13). <https://docs.rs/mockall>.
- **`proptest` + `tokio::test`** — property-based testing for stateful async logic. Wrap the property body in a runtime block:

  ```rust
  proptest! {
      #[test]
      fn prop_my_async_thing(input in any::<MyInput>()) {
          let rt = tokio::runtime::Builder::new_current_thread().enable_all().build()?;
          rt.block_on(async { assert!(check(input).await); });
      }
  }
  ```

- **`rstest`** — fixture + parameterised tests, works with `#[tokio::test]` via `#[rstest]` + `#[tokio::test]` stack. <https://docs.rs/rstest>.
- **`tokio-test`** — `assert_pending!`, `assert_ready!`, `task::spawn` mock — low-level utilities for testing custom `Future` impls. <https://docs.rs/tokio-test>.
- **Coverage:** `cargo tarpaulin` works with `#[tokio::test]` out of the box.
- **CI determinism:** run tests with `RUST_TEST_THREADS=1` if global state is involved, but prefer designing tests to be thread-safe (mock servers on random ports, scoped temp dirs).

## Tracing & observability

- **`tracing`** is the structured, span-based logging framework. Spans carry context across `.await` points — `log` / `slog` macros do not. <https://docs.rs/tracing>.
- **`#[instrument]`** macro auto-adds an entry/exit span around an `async fn`, with all arguments captured as fields. `#[instrument(skip(large_arg), fields(user_id = %req.user_id))]` to control what's recorded.
- **Log macros:** `error!`, `warn!`, `info!`, `debug!`, `trace!` — all accept structured KV fields (`info!(?req, count = items.len(), "received batch")`). Prefer named fields over format strings — they're machine-queryable downstream.
- **`tracing-subscriber`** assembles a `Subscriber`: `fmt` (text/json formatter), `EnvFilter` (`RUST_LOG=info,sqlx=warn,my_crate=debug`), `Registry` (composable layers). <https://docs.rs/tracing-subscriber>.
- **JSON output for production:** `tracing_subscriber::fmt().json().init()` or via the `bunyan` / `tracing-bunyan-formatter` crate.
- **`tracing-opentelemetry`** bridges `tracing` spans into OpenTelemetry's OTLP exporter for distributed tracing across services (Jaeger, Honeycomb, Datadog, Grafana Tempo). As of May 2026 the stack is `tracing 0.1` + `tracing-subscriber 0.3` + `tracing-opentelemetry 0.23` + `opentelemetry 0.22` + `opentelemetry_sdk 0.22` + `opentelemetry-otlp 0.15`. <https://docs.rs/tracing-opentelemetry>.
- **Context propagation across `tokio::spawn`:** every spawn detaches from the current span. Re-attach via `.instrument(span)`:

  ```rust
  use tracing::Instrument;
  let span = tracing::info_span!("request", request_id = %id);
  tokio::spawn(async move { handler().await }.instrument(span));
  ```

  Without `.instrument(...)`, logs inside the spawned task lose the `request_id` field and you cannot correlate to the parent request.

- **Subscriber initialisation pattern** (production-grade):

  ```rust
  use tracing_subscriber::{fmt, EnvFilter, prelude::*};
  fn init_tracing() {
      let fmt_layer = fmt::layer().json().with_target(true);
      let filter = EnvFilter::try_from_default_env()
          .unwrap_or_else(|_| EnvFilter::new("info,sqlx=warn"));
      tracing_subscriber::registry()
          .with(filter)
          .with(fmt_layer)
          // .with(OpenTelemetryLayer::new(tracer)) for OTLP export
          .init();
  }
  ```

- **`tokio-console`** — runtime introspection. Requires `console-subscriber = "0.4"` in the application and `RUSTFLAGS="--cfg tokio_unstable"` at build time. Run `tokio-console` against `localhost:6669` to see:
  - Task list (busy vs idle vs awoken),
  - Per-task busy/idle time and total scheduled time,
  - Lock contention (when did `tokio::sync::Mutex::lock` wait, and for how long),
  - Polls per task (a task that polls thousands of times per second is in a hot loop).
    Reference: <https://github.com/tokio-rs/console>.
- **Runtime metrics for SRE dashboards:** `tokio::runtime::Handle::current().metrics()` exposes worker counts, busy time, queued tasks, blocking-pool depth. Wire into Prometheus via `metrics` crate + `metrics-exporter-prometheus`.
- **Anti-pattern:** `println!("debug: {x}")` in async code (sync I/O blocks the executor; cross-ref anti-patterns).
