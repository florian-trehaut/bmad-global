# Step 9: Generate CLAUDE.local.md

## STEP GOAL:

Synthesize all knowledge generated during bootstrap into a self-contained `CLAUDE.local.md` — a gitignored, personal project guide that Claude Code auto-loads every session. The output must be useful to a Claude session with zero knowledge of BMAD workflows.

## MANDATORY SEQUENCE

### 1. Backup Existing File

CLAUDE.local.md is **gitignored and cannot be recovered from version control**. Protecting existing content is mandatory.

```bash
if [ -f "{MAIN_PROJECT_ROOT}/CLAUDE.local.md" ]; then
  BACKUP="{MAIN_PROJECT_ROOT}/CLAUDE.local.md.bak.$(date +%Y-%m-%d-%H%M%S)"
  cp "{MAIN_PROJECT_ROOT}/CLAUDE.local.md" "$BACKUP"
  echo "Backed up to: $BACKUP"
fi
```

If a backup was created, inform the user: "Existing CLAUDE.local.md backed up to `{backup_path}`."

### 2. Read Inputs

Read ALL of the following:

**A. Project configuration:**
- `{MAIN_PROJECT_ROOT}/.claude/workflow-context.md` — full file (frontmatter + body)

**B. Existing CLAUDE.md (if present):**
- `{MAIN_PROJECT_ROOT}/CLAUDE.md` — for coverage analysis (step 3)

**C. Approved knowledge files:**

Read each file that exists in `{MAIN_PROJECT_ROOT}/.claude/workflow-knowledge/`:
- `stack.md`
- `conventions.md`
- `domain-glossary.md`
- `review-perspectives.md`
- `investigation-checklist.md`
- `infrastructure.md`
- `tracker.md`
- `environment-config.md`
- `api-surface.md`
- `validation.md`
- `comm-platform.md`

Skip files that do not exist — they were rejected or not generated.

**D. Template:**
- `../templates/claude-local-template.md`

### 3. Analyze CLAUDE.md Coverage

**Goal:** Avoid duplicating what the project's shared CLAUDE.md already documents.

If `{MAIN_PROJECT_ROOT}/CLAUDE.md` exists:

Scan its headings (## and ###) and build a set `COVERED_BY_CLAUDE_MD` by detecting:

| Coverage area | Detection: heading contains (case-insensitive) |
|---------------|-----------------------------------------------|
| `architecture` | "architecture", "structure", "stack" |
| `conventions` | "convention", "style", "commit", "formatting" |
| `tests` | "test" |
| `commands` | "command", "script", "usage" |

If `{MAIN_PROJECT_ROOT}/CLAUDE.md` does **not** exist: `COVERED_BY_CLAUDE_MD` is empty — include all sections.

### 4. Compute Source Hash

Compute a staleness-tracking hash from all input files:

```bash
cat {MAIN_PROJECT_ROOT}/.claude/workflow-context.md {MAIN_PROJECT_ROOT}/.claude/workflow-knowledge/*.md 2>/dev/null | md5 | cut -c1-8
```

### 5. Synthesize CLAUDE.local.md

Load the template. For each section, apply the rules below. Remove all HTML comment placeholders after population.

**Prepend generation metadata** (invisible to Claude Code context loading):

```markdown
<!-- Generated by bmad-knowledge-bootstrap | {YYYY-MM-DD} | source_hash: {hash} -->
```

#### Section: Project Quick Reference (ALWAYS)

From workflow-context.md frontmatter, format as:

```markdown
## Project Quick Reference

- **Name**: {project_name} ({app_type})
- **Tracker**: {tracker} | **Forge**: {forge} ({forge_cli})
- **Quality gate**: `{quality_gate}`
- **Package manager**: {package_manager}
```

#### Section: Stack & Architecture (CONDITIONAL)

**Skip if** `architecture` is in `COVERED_BY_CLAUDE_MD`.

From `stack.md`, extract and condense:
- Project nature: one paragraph
- Architecture patterns: table (Scope | Pattern | Key Rules) — keep all rows
- Current stack: table (Component | Technology) — top 6 entries only
- Source file patterns: table (Pattern | Description)

#### Section: Code Conventions (CONDITIONAL)

**Skip if** `conventions` is in `COVERED_BY_CLAUDE_MD`.

From `conventions.md`, extract rules only (no explanations):
- Commit format: `type(scope): description` with allowed types as comma-separated inline list
- Branch strategy: one line with format
- Naming conventions: one line per category (skills, files, YAML, etc.)
- Import order: one line summary

#### Section: Test Rules (CONDITIONAL)

**Skip if** `tests` is in `COVERED_BY_CLAUDE_MD`.

From `stack.md` test section:
- Forbidden patterns: bullet list
- Test pyramid: compact table (Type | Framework | Suffix | Location)
- Test commands: code block

#### Section: Domain Vocabulary (ALWAYS)

From `domain-glossary.md`, select the **10-15 most important terms** — those that actively affect how Claude should write code or understand the project.

Format as: `**Term** — one-line definition`

If domain-glossary.md does not exist or has no real content: replace section with "No domain-specific vocabulary documented for this project."

#### Section: Review Checklist (ALWAYS)

From `review-perspectives.md`, condense each perspective to **2-3 actionable items**.

Format as grouped bullets:

```markdown
- **Perspective name**:
  - Key check 1
  - Key check 2
```

Include severity classification (one line per level).

If review-perspectives.md does not exist: replace section with "No review perspectives documented for this project."

#### Section: Investigation Guides (CONDITIONAL)

**Skip if** `investigation-checklist.md` does not exist or contains only "No X detected" placeholders.

From `investigation-checklist.md`: condense key investigation points per domain area. Include file:line references where available.

#### Section: Personal Preferences (ALWAYS)

From workflow-context.md frontmatter:

```markdown
## Personal Preferences

- **Communication language**: {communication_language}
- **Document output language**: {document_output_language}
- **User**: {user_name} ({user_skill_level})
```

#### Section: Workflow Integration (ALWAYS)

From workflow-context.md frontmatter:

- **Worktree templates** (only if worktree_enabled or worktree templates exist): format as `name: pattern` list
- **Branch template**: `{branch_template}`
- **Forge commands**: PR create, PR list (from forge_mr_create, forge_mr_list)
- **Tracker states**: `{tracker}` with states listed inline
- **Quality gate**: `{quality_gate}` command

#### Section: Available Skills (ALWAYS)

Scan `~/.claude/skills/bmad-*/SKILL.md`:
- Read `name` and `description` from each SKILL.md frontmatter
- Exclude utility skills: `bmad-shared`, `bmad-help`
- Format as: `**/skill-name** — first sentence of description`
- If >10 skills, group by purpose (analysis, planning, implementation, review)

If no skills found: replace with "No BMAD skills installed. Run `npx @bmad-code-org/bmad-method install` to install."

#### Section: Deep Knowledge (ALWAYS)

List only files that **actually exist** in `{MAIN_PROJECT_ROOT}/.claude/workflow-knowledge/`.

Format as:

```markdown
## Deep Knowledge

When you need detailed information beyond this summary, read files from `.claude/workflow-knowledge/`:

- `stack.md` — Full tech stack, architecture details, test rules
- `conventions.md` — Complete code style, PR standards, naming rules
- ...
```

Use the "Content" column from workflow-context.md's Knowledge Files table if available, otherwise derive one-line descriptions from the file headings.

### 6. Self-Containment Verification

Before presenting the draft, verify:

- [ ] No references to "workflow-context.md" as a concept (only as a file path in Deep Knowledge)
- [ ] No BMAD-specific jargon without explanation (e.g., "JIT loading", "step-file architecture", "HALT checkpoint")
- [ ] No instructions that assume BMAD skill awareness (e.g., "the workflow will load this")
- [ ] The Deep Knowledge section explains files as "read this file for more details", not "this is loaded by bmad-dev-story"
- [ ] All skill names in "Available Skills" are presented as slash commands a user can invoke

If any violation is found, rewrite the offending section before presenting.

### 7. Present for Review

Present the COMPLETE generated content, then:

> **[A]** Accept — write as-is
> **[E]** Edit — modify specific sections
> **[R]** Reject — skip CLAUDE.local.md generation

**Menu handling:**

- **A**: Proceed to write.
- **E**: Ask user what to edit. Apply corrections. Re-present for confirmation. Loop until A or R.
- **R**: Log reason. Inform user: "Skipped. You can regenerate later by running the bootstrap again." Do NOT write the file.

### 8. Write File

Write to `{MAIN_PROJECT_ROOT}/CLAUDE.local.md`.

Log: "Written: CLAUDE.local.md ({N} lines)"

**Gitignore check:**

```bash
grep -qE "^CLAUDE\.local\.md$|^CLAUDE\.local\.md\b" {MAIN_PROJECT_ROOT}/.gitignore 2>/dev/null
```

If NOT found in .gitignore: warn the user — "CLAUDE.local.md is not in `.gitignore`. This file contains personal preferences and should not be committed. Consider adding `CLAUDE.local.md` to `.gitignore`."

**Worktree notice** (if running from a worktree): "CLAUDE.local.md was written to the main repository root at `{MAIN_PROJECT_ROOT}/`, not the worktree."

## SYSTEM SUCCESS/FAILURE METRICS

### SUCCESS:

- Existing CLAUDE.local.md backed up before any write
- CLAUDE.md analyzed for coverage (if exists)
- Conditional sections correctly skipped when covered
- Output is self-contained — no unexplained BMAD jargon
- User explicitly approved before writing
- Gitignore status checked and warned if missing

### FAILURE:

- Overwriting CLAUDE.local.md without backup
- Duplicating content already in CLAUDE.md
- Writing without user review
- Using BMAD workflow terminology without context
- Referencing knowledge files as "loaded by workflow X" instead of "read this file for details"
