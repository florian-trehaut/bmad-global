# Step 3: Validate and Summarize

## STEP GOAL

Run all generated tests, fix any failures, and produce a coverage summary with concrete metrics.

## RULES

- Run ALL generated tests — never skip execution
- Fix failures immediately — do not defer to the user unless the root cause is ambiguous
- After fixing, re-run to confirm the fix works
- Coverage metrics must reflect actual execution results, not estimates
- Summary must be saved to a file in the project

## SEQUENCE

### 1. Run tests

Execute tests using `{TEST_RUNNER_CMD}` detected in Step 1.

If API and E2E tests use different runners, run them separately:
1. API tests first
2. E2E tests second

Record results:
- `PASS_COUNT` = number of passing test cases
- `FAIL_COUNT` = number of failing test cases

### 2. Fix failures

For each failing test:

1. Read the error output carefully
2. Determine root cause:
   - **Test bug** (wrong assertion, bad selector, timing issue) — fix the test
   - **Missing setup** (database not seeded, service not running) — add setup or skip with clear reason
   - **Actual application bug** — do NOT fix the application code. Flag it:
     > "Application bug found: {description}. Test expectation is correct but the application returns {actual}. Keeping the test as-is — it will pass once the bug is fixed."

3. After fixing test bugs, re-run the failing tests to confirm they pass

4. Update `PASS_COUNT` and `FAIL_COUNT` after fixes.

### 3. Create summary

Build the test automation summary:

```markdown
# Test Automation Summary

**Date:** {current date}
**Framework:** {TEST_FRAMEWORK}
**Generated by:** Quinn (bmad-qa-automate)

## Generated Tests

### API Tests
- [x] {path/to/test-file} — {description} ({N} cases)
- ...

### E2E Tests
- [x] {path/to/test-file} — {description} ({N} cases)
- ...

## Results

| Metric | Count |
|--------|-------|
| Total test cases | {API_TEST_COUNT + E2E_TEST_COUNT} |
| Passing | {PASS_COUNT} |
| Failing | {FAIL_COUNT} |
| Skipped | {SKIP_COUNT} |

## Coverage

- API endpoints: {covered}/{total} covered
- UI features: {covered}/{total} covered

## Application Bugs Found

{List any application bugs discovered during testing, or "None"}

## Next Steps

- Run tests in CI
- Add more edge cases as needed
- For advanced testing (risk-based strategy, quality gates, NFR assessment): use `bmad-test-design` skill
```

### 4. Save summary

Save the summary to `{project-root}/tests/test-summary.md`.

If the `tests/` directory doesn't exist, save to the project's existing test directory root. If no test directory exists, save to `{project-root}/test-summary.md`.

### 5. Present results

> **Tests generated and validated.**
>
> - **Total:** {total} test cases ({PASS_COUNT} passing, {FAIL_COUNT} failing)
> - **API:** {API_TEST_COUNT} cases across {N} files
> - **E2E:** {E2E_TEST_COUNT} cases across {N} files
> - **Summary saved to:** `{summary path}`
>
> {If FAIL_COUNT > 0: list remaining failures with explanations}
