# Domain Glossary — Knowledge

## Ubiquitous Language

| Term | Definition | Code Representation |
|---|---|---|
| **Module** | A distributable package of skills grouped by purpose. Has a code, name, config variables, and directory creation rules. | `module.yaml` in each module root (`src/bmm-skills/`, `src/core-skills/`) |
| **Skill** | The atomic unit of AI capability. A self-contained directory with SKILL.md entrypoint, optional workflow/steps/data/templates. | `bmad-{name}/SKILL.md` — frontmatter: `name`, `description` |
| **Agent** | A persona-bearing skill (prefixed `bmad-agent-*`) embodying a named character with communication style, principles, and capability menu. | `bmad-agent-{name}/SKILL.md` — body contains persona definition |
| **Workflow** | The procedural logic of a skill, defining initialization, role, step loading, and execution rules. | `workflow.md` in skill directory |
| **Step** | A discrete execution unit within a workflow, loaded JIT one at a time. | `steps/step-NN-description.md` (2-10 per skill) |
| **Phase** | A numbered lifecycle stage grouping skills: 1-analysis, 2-plan-workflows, 3-solutioning, 4-implementation. | Directory name under `src/bmm-skills/` |
| **Template** | A fill-in-the-blank output scaffold used by steps to generate artifacts. | Files in `templates/` within a skill |
| **Data** | Reference material loaded JIT by steps (checklists, classification rules, knowledge). Not user-facing output. | Files in `data/` within a skill |
| **Subagent Workflow** | A workflow delegated to a spawned sub-agent for parallel investigation. | Files in `subagent-workflows/` within a skill |
| **Knowledge** | Long-lived project context files generated by bmad-knowledge-bootstrap. | `.claude/workflow-knowledge/*.md` and `.claude/workflow-context.md` |
| **Shared Rules** | Common rules loaded by all bmad-* workflow skills (zero-fallback, spawn protocol, team router). | `bmad-shared/` in core-skills |
| **Persona** | The character identity in an agent SKILL.md body (name, overview, style, principles, critical actions). | Body content of `bmad-agent-*/SKILL.md` |
| **HALT** | A mandatory stop-and-report directive — AI must stop execution, display message, and wait for user action. | Keyword in workflow/step instructions |
| **Config Variable** | A user-configurable value defined in module.yaml with prompt, default, and result interpolation. | `module.yaml` variable entries |

## Bounded Contexts

### Context: Skill Content

**Responsibility:** The core product — Markdown/YAML skills, workflows, step files, templates, and data that LLMs execute.

**Key Entities:**
- Skill (SKILL.md + workflow.md + steps/)
- Module (module.yaml + skills grouped by phase)
- Agent (persona-bearing skill with capability menu)
- Template (output scaffold)
- Data (reference material)

**Boundaries:** Does NOT handle installation, validation, or distribution — those are separate contexts.

### Context: CLI / Installer

**Responsibility:** Installing skills from `src/` into user projects (`~/.claude/skills/`), managing config, updates.

**Key Entities:**
- GlobalInstaller (core install logic)
- ModuleManager (reads module.yaml, resolves variables)
- Commands (install, uninstall, status)

**Boundaries:** Does NOT modify skill content. Reads module.yaml and SKILL.md, copies files, resolves variables.

### Context: Validation

**Key Entities:**

- FileRefValidator (cross-file reference integrity)
- TestSuite (Node.js test scripts)

**Boundaries:** Read-only analysis. Does NOT modify skills. Reports findings with file:line references.

### Context: Documentation

**Key Entities:**

## Entity Relationships

- **Module** contains 1..N **Skills** (organized by phase)
- **Skill** contains 1 **SKILL.md** + optional **workflow.md** + 0..10 **Steps**
- **Skill** may contain **Data**, **Templates**, **Subagent Workflows**
- **Agent** is a specialized **Skill** with a **Persona** in its body
- **Installer** reads **Module** config and copies **Skills** to target
- **Validator** reads **Skills** and reports findings

## Domain Rules

- Every skill must have a `name` matching its directory name (SKILL-05)
- Skill names must follow `bmad-*` kebab-case pattern (SKILL-04)
- Skills must have 2-10 step files if steps/ directory exists (STEP-07)
- Step files must be loaded JIT — no forward loading (STEP-05)
- Missing required values must HALT, never silently default
- Module.yaml variables use `{value}`, `{project-root}`, `{directory_name}` interpolation
